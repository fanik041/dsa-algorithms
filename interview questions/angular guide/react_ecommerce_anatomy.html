Component Tree

React apps are a tree of components. App is the root. Each component renders its children. Data flows DOWN via props, events flow UP via callbacks. The tree determines rendering order and re-render boundaries.

Redux Store

Global state container. Three slices (products, cart, auth) manage domain state. Components read via useSelector, write via useDispatch. Predictable: action → reducer → new state → UI update.

React Query

Manages SERVER state (API data). Handles caching, refetching, loading/error states automatically. Separate from Redux which manages CLIENT state. Together they cover all state needs.

React Router

Maps URLs to components. <Routes> defines the route tree. <Outlet> renders child routes (like a layout slot). Nested routes enable shared layouts.

Provider Wrapping Order

QueryClientProvider (outermost) → Redux Provider → BrowserRouter → App content. Each provider makes its context available to all descendants. Order matters for dependency chains.

File Structure

src/components/ (shared UI), src/features/ (domain: products, cart, auth), src/hooks/ (custom hooks), src/store/ (Redux slices), src/api/ (API functions), src/types/ (TypeScript interfaces).

JSX is Syntactic Sugar

JSX compiles to React.createElement() calls. It's not HTML — it's JavaScript expressions that produce React elements.

This is why you can use JS inside {} and why className instead of class (class is a JS reserved word).
Expressions in JSX

Anything in {} is a JavaScript expression. Variables, function calls, ternaries, template literals.

NOT statements (no if/for/while). This constraint pushes you toward expression-based patterns like ternary and .map().
Conditional Rendering

Three patterns: (1) Ternary for either/or, (2) && for show/hide, (3) early return for guard clauses.

Watch out: {0 && <Component />} renders "0" because 0 is falsy but not false. Always use {count > 0 && ...}.
Keys in Lists

Keys help React identify which items changed, were added, or removed. Must be unique among siblings and stable across re-renders.

NEVER use array index for dynamic lists — it causes bugs with reordering and component state.
Fragments

<>...</> groups elements without adding a DOM node. Use <React.Fragment key={id}> when you need a key (e.g., in .map()).

Keeps the DOM clean and avoids CSS layout issues from extra wrapper divs.
JSX vs HTML

className not class, htmlFor not for, camelCase events (onClick), style takes an object not a string, boolean attributes differ (disabled not disabled="disabled"), self-closing tags required (<img /> not <img>).

Functional Components

The only component type you need. Functions that accept props and return JSX. Class components are legacy.

Always prefer function components with hooks.
Props & Destructuring

Props are read-only inputs. Destructure in the function signature for clarity. Use TypeScript interfaces for type safety.

Optional props use ?, defaults use = value in destructuring.
Children Prop

children is a special prop — anything between opening/closing tags. Type it as ReactNode (broadest) or ReactElement (JSX only).

Enables the slot pattern for flexible component composition.
Prop Spreading

...rest collects remaining props. {...rest} spreads them onto an element. Extend ComponentPropsWithoutRef<'button'> to accept all native button attributes.

Powerful but use carefully — can pass unintended props.
Composition Over Inheritance

React has NO inheritance pattern. Build complex UI by composing simpler components. Slots via children/props.

Specialization via wrapping (IconButton wraps Button). This is a fundamental React principle.
When to Split Components

Split when: reused, has own state, complex logic, needs testing, distinct concept.

Don't split too early — premature abstraction is worse than duplication. Rule of thumb: if a component is >100 lines, consider splitting.

type CartAction =
  | { type: 'ADD_ITEM'; payload: Product }
  | { type: 'REMOVE_ITEM'; payload: number }
  | { type: 'UPDATE_QTY';
      payload: { id: number; qty: number } }
  | { type: 'CLEAR' };

function cartReducer(
  state: CartItem[],
  action: CartAction
): CartItem[] {
  switch (action.type) {
    case 'ADD_ITEM': {
      const exists = state.find(
        i => i.id === action.payload.id);
      if (exists) return state.map(i =>
        i.id === action.payload.id
          ? { ...i, qty: i.qty + 1 }
          : i
      );
      return [...state,
        { ...action.payload, qty: 1 }];
    }
    case 'REMOVE_ITEM':
      return state.filter(
        i => i.id !== action.payload);
    case 'UPDATE_QTY':
      return state.map(i =>
        i.id === action.payload.id
          ? { ...i, qty: action.payload.qty }
          : i
      );
    case 'CLEAR':
      return [];
  }
}

useState — The most basic hook. const [value, setValue] = useState(initial). Returns current value + setter function. Calling setter triggers re-render. Initial value only used on first render. Use lazy initializer () => expensive() for costly defaults.
Updater Function — setState(prev => prev + 1) guarantees you work with the latest state. setState(count + 1) may use a stale closure. ALWAYS use updater when new state depends on old state. Especially important when calling setState multiple times.
Automatic Batching (React 18) — Multiple setState calls in the same event handler cause only ONE re-render. React 18 extends this to setTimeout, Promises, and native events too. Previously only event handlers were batched. Use flushSync() to opt out (rare).
useReducer — Like useState but for complex state logic. Takes a reducer function + initial state. Returns [state, dispatch]. dispatch({type, payload}) sends actions. Better when: next state depends on previous, multiple sub-values, complex update logic.
Immutable Updates — React detects changes by reference (===). Mutating state directly (push, splice, direct assignment) won't trigger re-renders. Always create new objects/arrays with spread operator. This is WHY Redux Toolkit uses Immer.
Lifting State Up — When siblings need to share state, move it to their closest common parent. Parent owns the state, passes down via props. This is React's answer to "how do components communicate?" For distant components, use Context or Redux instead.

  // Runs on MOUNT only (empty deps)
  useEffect(() => {
    console.log('Component mounted');

Effect Lifecycle — useEffect runs AFTER render (not during). Mount: first render. Update: when deps change. Cleanup: before next run and on unmount. Think of it as: "after React updates the DOM, do this side effect."
Dependency Array — Empty [] = mount only. [dep] = mount + when dep changes. No array = every render (almost never want this). React compares deps by reference (===). Objects/arrays create new references each render — use useMemo or move inside effect.

useRef

Two uses: (1) DOM access — attach to elements via ref={myRef}, access via .current. (2) Mutable container — store values that persist across renders WITHOUT causing re-renders.

Like an instance variable. Great for timers, previous values, WebSocket refs.
useMemo

Caches the RESULT of an expensive computation. Only recomputes when dependencies change. Use for: sorting/filtering large arrays, complex calculations, creating objects used as effect deps.

Don't overuse — has its own memory cost.
useCallback

Caches a FUNCTION reference. Same function identity between renders when deps haven't changed. Essential when passing callbacks to React.memo children — without it, memo is useless because the function is "new" every render.
useId

Generates unique IDs that are stable between server and client renders. Use for form labels, ARIA attributes.

Never use Math.random() or incrementing counters — they cause hydration mismatches in SSR.
useImperativeHandle

Customizes what a ref exposes to the parent. Instead of exposing the raw DOM node, expose a clean API ({ focus, clear, scrollTo }). Used with forwardRef.

Keeps component encapsulation intact.
Rules of Hooks

(1) Only at top level — no conditions, loops, or nesting. (2) Only in React functions — components or custom hooks. React tracks hooks by CALL ORDER. Breaking order = bugs.

The ESLint plugin react-hooks/rules-of-hooks enforces this.

function useDebounce<T>(value: T, delay: number): T {
  const [debounced, setDebounced] = useState(value);

  useEffect(() => {
    const timer = setTimeout(
      () => setDebounced(value), delay);
    return () => clearTimeout(timer);
  }, [value, delay]);

  return debounced;
}

function useLocalStorage<T>(
  key: string,
  initialValue: T
): [T, (value: T) => void] {
  const [stored, setStored] = useState<T>(() => {
    try {
      const item = localStorage.getItem(key);
      return item ? JSON.parse(item) : initialValue;
    } catch { return initialValue; }
  });

  const setValue = (value: T) => {
    setStored(value);
    localStorage.setItem(key, JSON.stringify(value));
  };

  return [stored, setValue];
}

function useFetch<T>(url: string) {
  const [data, setData] = useState<T | null>(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState<Error | null>(null);

  useEffect(() => {
    let cancelled = false;
    setLoading(true);
    setError(null);

    fetch(url)
      .then(res => {
        if (!res.ok) throw new Error(res.statusText);
        return res.json();
      })
      .then(data => {
        if (!cancelled) {
          setData(data);
          setLoading(false);
        }
      })
      .catch(err => {
        if (!cancelled) {
          setError(err);
          setLoading(false);
        }
      });
    return () => { cancelled = true; };
  }, [url]);

  return { data, loading, error };
}

function useOnClickOutside(
  ref: RefObject<HTMLElement>,
  handler: () => void
) {
  useEffect(() => {
    const listener = (e: MouseEvent | TouchEvent) => {
      if (!ref.current ||
          ref.current.contains(e.target as Node))
        return;
      handler();
    };
    document.addEventListener('mousedown', listener);
    document.addEventListener('touchstart', listener);
    return () => {
      document.removeEventListener('mousedown', listener);
      document.removeEventListener('touchstart', listener);
    };
  }, [ref, handler]);
}

function useMediaQuery(query: string): boolean {
  const [matches, setMatches] = useState(
    () => window.matchMedia(query).matches
  );
  useEffect(() => {
    const mql = window.matchMedia(query);
    const handler = (e: MediaQueryListEvent) =>
      setMatches(e.matches);
    mql.addEventListener('change', handler);
    return () =>
      mql.removeEventListener('change', handler);
  }, [query]);
  return matches;
}

Custom Hook Pattern

Extract reusable stateful logic into functions prefixed with use. They can call other hooks. Return values as tuples [value, setter] or objects { data, loading }. Compose hooks together for complex behavior.

useDebounce

Delays updating a value until the user stops changing it. Perfect for search inputs — avoid an API call on every keystroke. Uses useEffect + setTimeout internally. Returns the debounced value.

useLocalStorage

Syncs state with localStorage. Lazy initialization reads from storage on first render. Updates both React state and localStorage on change. Survives page refresh. Generic type parameter for type safety.

useFetch

Encapsulates fetch lifecycle (loading, data, error). Handles race conditions with a cancelled flag. Good for learning, but use React Query for production — it adds caching, retry, refetching, and deduplication.

useOnClickOutside

Detects clicks outside a referenced element. Perfect for dropdowns, modals, popovers. Listens on mousedown/touchstart. Cleans up listeners on unmount. Common real-world custom hook.

useMediaQuery

Subscribes to CSS media query changes. Returns boolean match status. Reactive — updates when screen size changes. Useful for responsive behavior that can’t be done with CSS alone.

Naming & Composition

Always prefix with use — tells React it’s a hook. Return consistent shapes (tuple for simple, object for complex). Hooks can call other hooks — compose useDebounce inside useFetch for debounced search. Keep hooks focused on one concern.

  { items: [], total: 0 };

  name: 'cart',
  initialState,
  reducers: {
    addItem(state, action: PayloadAction<Product>) {
      // Immer lets you "mutate" —
      // it's actually immutable under the hood!
      const existing = state.items.find(
        i => i.id === action.payload.id);
      if (existing) {
        existing.qty += 1;  // OK with Immer!
      } else {
        state.items.push(
          { ...action.payload, qty: 1 });
      }
      state.total = state.items.reduce(
        (sum, i) => sum + i.price * i.qty, 0);
    },
    removeItem(state, action: PayloadAction<number>) {
      state.items = state.items.filter(
        i => i.id !== action.payload);
      state.total = state.items.reduce(
        (sum, i) => sum + i.price * i.qty, 0);
    },
    clearCart(state) {
      state.items = [];
      state.total = 0;
    },
  },
});

import { createAsyncThunk } from '@reduxjs/toolkit';

export const fetchProducts = createAsyncThunk(
  'products/fetchAll',
  async (
    params: { category?: string },
    { rejectWithValue }
  ) => {
    try {
      const res = await fetch(
        `/api/products?category=${params.category ?? ''}`
      );
      if (!res.ok) throw new Error('Failed');
      return (await res.json()) as Product[];
    } catch (err) {
      return rejectWithValue(
        (err as Error).message
      );
    }
  }
);

// Dispatch like any action:
dispatch(fetchProducts({ category: 'shoes' }));

// The thunk dispatches 3 actions automatically:
// 1. products/fetchAll/pending
// 2. products/fetchAll/fulfilled  (on success)
// 3. products/fetchAll/rejected   (on error)

const productSlice = createSlice({
  name: 'products',
  initialState: {
    items: [] as Product[],
    status: 'idle' as
      'idle' | 'loading' | 'succeeded' | 'failed',
    error: null as string | null,
  },
  reducers: {},
  extraReducers: (builder) => {
    builder
      .addCase(fetchProducts.pending, (state) => {
        state.status = 'loading';
        state.error = null;
      })
      .addCase(fetchProducts.fulfilled,
        (state, action) => {
          state.status = 'succeeded';
          state.items = action.payload;
        }
      )
      .addCase(fetchProducts.rejected,
        (state, action) => {
          state.status = 'failed';
          state.error = action.payload as string;
        }
      );
  },
});

useQuery

Fetches and caches data. Takes queryKey (cache identity) and queryFn (fetch function). Returns { data, isLoading, isError, error, refetch }. Automatically refetches on window focus, mount, and stale time.

Query Keys

Array that uniquely identifies cached data. ['products'] vs ['products', 'shoes'] are different cache entries. Include all variables the query depends on. React Query auto-refetches when key changes.

Stale Time & GC Time

staleTime: how long data is "fresh" (no refetch). gcTime (was cacheTime): how long unused cache stays in memory. Fresh data serves instantly from cache. Stale data serves from cache but refetches in background.

useMutation

For creating/updating/deleting data. mutate() triggers the mutation. Returns isPending, isError, isSuccess. Use onSuccess to invalidate related queries. Unlike queries, mutations don't run automatically.

Cache Invalidation

queryClient.invalidateQueries({ queryKey: ['cart'] }) marks cache as stale and triggers refetch. Call after mutations to keep UI in sync. Can invalidate exact keys or prefix-match.

Optimistic Updates

Update UI immediately, before server confirms. onMutate: cancel queries, snapshot old data, update cache. onError: rollback to snapshot. onSettled: refetch to ensure consistency. Makes UI feel instant.

Infinite Queries

useInfiniteQuery for paginated data. getNextPageParam extracts cursor from last page. fetchNextPage() loads more. Data is an array of pages. Perfect for infinite scroll or "Load More" buttons.

Dependent & Conditional Queries

enabled: false prevents auto-fetch. Use for: dependent queries (wait for parent data), conditional fetching (only when user types 3+ chars), manual trigger. Disabled queries don't refetch on window focus.

Controlled Inputs

React owns the input's value via value + onChange. Every keystroke triggers setState, causing a re-render. React is the single source of truth. Use for: real-time validation, conditional rendering based on input, enforcing input format (e.g., phone numbers).

Uncontrolled Inputs

The DOM owns the input's value. Use ref to read the value when needed (e.g., on submit). defaultValue sets initial value. Fewer re-renders, simpler code. Use for: simple forms, file inputs (always uncontrolled), when you only need the value on submit.

React Hook Form

Uses uncontrolled inputs internally for minimal re-renders. register() connects inputs. handleSubmit() validates before calling your function. Built-in validation, error messages, and form state (isSubmitting, isDirty, isValid). Best for complex forms with many fields.

Validation

React Hook Form supports: required, minLength, maxLength, pattern (regex), and custom validate functions. The errors object contains field-level error messages. Validation runs on submit by default, can be set to onChange or onBlur.

Form Submission

e.preventDefault() stops page reload. Track isSubmitting to disable the button and show loading. Handle errors with try/catch. On success, show confirmation or redirect. Always validate before submitting.

Debounced Search

Custom useDebounce hook delays value updates. Combined with React Query's enabled guard — only fetches when debounced query is 3+ characters. Prevents excessive API calls while typing. Pattern: type → wait 300ms → fetch.

Composition (Slots)

The default pattern in React. Pass children for main content, named props (header, footer) for slots. More flexible than inheritance. The parent decides WHAT to render, the component decides WHERE.

Compound Components

Related components that share implicit state via Context. Select provides state, Select.Option consumes it. API looks like HTML (<select><option>). Great for UI libraries. The parent controls state, children handle rendering.

Render Props

A component calls a function prop to let the consumer decide what to render. Provides inversion of control. Largely replaced by custom hooks which achieve the same reuse with cleaner syntax. Still seen in some libraries (e.g., Formik's <Field>).

Higher-Order Components

A function that takes a component and returns a new component with added behavior. withAuth(Checkout) adds auth checking. Legacy pattern — harder to debug (wrapped display names), can't use hooks. Prefer custom hooks or wrapper components today.

Headless Components

Provide logic without UI. Implemented as custom hooks (useToggle, useDropdown). The consumer provides all rendering. Maximum flexibility. Libraries like Headless UI and Radix use this pattern. The modern evolution of render props.

When to Use Each

Composition: default choice, most cases. Compound: related UI components (tabs, accordions). Custom hooks: reusable logic without UI. Render props: rare, library APIs. HOCs: legacy code only. Start simple, add patterns as complexity grows.

Error Boundaries

The ONLY way to catch render errors in React. Must be a class component — there is no hook equivalent. getDerivedStateFromError updates state to show fallback. componentDidCatch logs/reports. Does NOT catch: event handler errors, async errors, or server-side errors.

Fallback UI

What users see instead of a crash. Include an error message and a retry button. Use per-route boundaries so one section crashing doesn't take down the whole app. The boundary catches errors in its entire subtree.

try/catch in Effects

Error boundaries don't catch async errors. Use try/catch inside useEffect for API calls. Set error state for UI feedback. Use finally to always clean up loading state. Pattern: try fetch → catch set error → finally stop loading.

React Query Errors

Built-in retry with exponential backoff. isError and error object for UI. onError callback in mutations for side effects (redirect, toast). retryDelay calculates wait time. Combines perfectly with error boundaries via throwOnError.

Global Error Handler

Listen for unhandledrejection events to catch uncaught promise errors. event.preventDefault() prevents console error. Report to monitoring (Sentry, LogRocket). Last line of defense — specific error handling is always preferred.

Toast Notifications

Non-blocking error feedback for the user. notify('message', 'error') shows a temporary alert. Auto-dismiss after timeout. Don't use for critical errors that need user action — use inline error messages or modal dialogs instead.

function useElementSize(
  ref: RefObject<HTMLElement>
) {
  const [size, setSize] = useState(
    { width: 0, height: 0 });

  useEffect(() => {
    if (!ref.current) return;

    const observer = new ResizeObserver(
      ([entry]) => {
        setSize({
          width: entry.contentRect.width,
          height: entry.contentRect.height,
        });
      }
    );

    observer.observe(ref.current);
    return () => observer.disconnect();
  }, [ref]);

  return size;
}

useRef for DOM Access

useRef creates a mutable object with a .current property that persists across renders. Pass it to a JSX element's ref prop to get the DOM node. Common uses: focus management, scroll position, measuring elements. Changing .current does NOT trigger re-render.

Callback Refs

A function passed as ref instead of an object. Called with the DOM node on mount, null on unmount. More flexible than useRef — can measure elements on attach, trigger state updates, or handle dynamic lists of refs.

forwardRef

Passes a ref from parent through to a child's internal DOM element. Required for reusable components that need to expose their DOM. Without it, ref on a custom component returns nothing. Typed as forwardRef<ElementType, PropsType>.

useImperativeHandle

Customizes what the parent sees when using a ref on a child component. Instead of exposing the raw DOM node, expose a clean API: { play(), pause(), reset() }. Hides internal implementation. Always use with forwardRef.

createPortal

Renders children into a DOM node outside the parent hierarchy. Perfect for modals, tooltips, dropdowns that need to escape overflow: hidden or z-index stacking. Events still bubble through the React tree (not the DOM tree).

Measuring DOM Elements

ResizeObserver watches for size changes. getBoundingClientRect() gets position and dimensions. Wrap in a custom hook (useElementSize) for reuse. Clean up observer in effect return to prevent memory leaks.

React.memo

Wraps a component to skip re-rendering when props haven't changed (shallow comparison). Custom comparator for fine-grained control. Only works if the component is pure — same props = same output. Pair with useCallback for function props.

useMemo

Caches the result of an expensive computation. Only recalculates when dependencies change. Use for: sorting/filtering large arrays, complex calculations, creating objects/arrays used as effect dependencies. Don't use for cheap operations — the comparison itself has cost.

useCallback

Returns the same function reference across renders (when deps unchanged). Prevents React.memo children from re-rendering due to new function props. Essentially useMemo(() => fn, deps). Only useful when the child is memoized.

lazy + Suspense

React.lazy() creates a component that loads its code on demand via dynamic import(). Each lazy component becomes a separate JS chunk. Suspense shows a fallback while the chunk downloads. Best at route level for maximum impact.

Virtualization

Only renders items visible in the viewport. 10,000 items but only ~10 DOM nodes. Libraries: react-window (lightweight), react-virtuoso (feature-rich). Essential for long lists, tables, and grids. Dramatically reduces DOM node count and memory usage.

React DevTools Profiler

Records renders and shows flamegraphs per commit. "Highlight Updates" visually shows which components re-render. Identifies unnecessary re-renders and slow components. Use BEFORE optimizing — measure first, don't guess. Available as a browser extension.

When NOT to Memoize

Don't memoize cheap operations (string concatenation, simple math). useMemo/useCallback have memory and comparison overhead. Don't wrap in useCallback unless the child is React.memo'd. Rule: measure first with Profiler, then optimize the bottleneck.

useTransition

Marks state updates as non-urgent. startTransition(() => setState(...)) tells React this update can be interrupted. isPending lets you show a loading indicator. The old UI stays visible until the new one is ready — no blank screen.

useDeferredValue

Returns a "lagging" version of a value. React updates it in the background during idle time. Compare query !== deferredQuery to detect staleness. Like debounce but React-managed — adapts to device speed. Use when you receive a fast-changing value from a parent.

Suspense

Declarative loading states. Shows fallback while children are "suspended" (loading code or data). Works with React.lazy for code splitting and useSuspenseQuery for data. Nest multiple boundaries for granular loading states.

Automatic Batching

React 18 batches ALL state updates into a single re-render — even in setTimeout, promises, and native event handlers. React 17 only batched inside React event handlers. This means fewer renders and better performance by default.

Urgent vs Transition Updates

Urgent: typing, clicking — must feel instant. Transition: filtering, tab switching — can wait. React can interrupt a transition update if an urgent update arrives. This keeps the UI responsive even during expensive renders.

flushSync

Forces React to flush updates synchronously. Opts out of batching for specific updates. Rare use case: when you need to read from the DOM immediately after a state update (e.g., scroll position, focus management). Avoid unless absolutely necessary.

Server Components

Run on the server only — their code is NEVER sent to the browser. Can directly access databases, files, and environment variables. Cannot use hooks, state, or browser APIs. Default in Next.js App Router. Great for data fetching and static content.

Server Actions

The process of attaching event handlers to server-rendered HTML. Hydration mismatch: when server HTML differs from client render (e.g., timestamps, random values). Fix: use useEffect for client-only values so initial render matches server.

Streaming SSR

Server sends HTML in chunks instead of waiting for everything. <Suspense> boundaries mark where streaming breaks occur. Shell (layout) sends first, then each Suspense boundary fills in as its data resolves. Faster Time to First Byte (TTFB).

Server vs Client Decision

Server: data fetching, static content, sensitive logic, large dependencies. Client: interactivity, state, effects, browser APIs, real-time updates. Default is server in App Router. Only add 'use client' when you need interactivity.

Testing Library Philosophy

"The more your tests resemble the way your software is used, the more confidence they give you." Query by role, label, and text — NOT by CSS class or test ID. Test behavior (what the user sees and does), not implementation details (state, props, hooks).

render & screen

render(<Component />) renders into a virtual DOM. screen provides queries: getBy (throws if missing), queryBy (returns null), findBy (waits, returns promise). within(element) scopes queries to a container.

userEvent vs fireEvent

userEvent.setup() simulates realistic user interactions (focus → keydown → keyup → input → change). fireEvent dispatches a single DOM event. Always prefer userEvent — it catches more bugs by simulating the full interaction chain.

Async Testing

findBy* returns a promise — waits up to 1s for element to appear. waitFor(() => expect(...)) retries an assertion until it passes. queryBy* returns null instead of throwing — use to assert element is NOT present.

Mocking with MSW

Mock Service Worker intercepts requests at the network level — your code doesn't know it's mocked. setupServer(...handlers) defines default responses. server.use() overrides handlers per-test (e.g., simulate errors). More realistic than mocking fetch/axios.

Testing Hooks

renderHook renders a hook in isolation. result.current holds the return value. rerender() simulates prop changes. Wrap state updates in act() to flush React updates. Test custom hooks separately from components for cleaner tests.

Testing with Redux

Create a test store with preloadedState. Wrap component in <Provider>. Test the component's behavior (rendered output, user interactions) — not the store internals. For complex setups, create a renderWithProviders utility.

Build-Time Optimizations

Code splitting (HIGH): React.lazy() creates separate JS chunks loaded on demand. Split at route level for maximum impact. Tree shaking (HIGH): use named imports so bundlers can remove unused code. Bundle analysis (MEDIUM): use source-map-explorer to find bloat.

Runtime Optimizations

React.memo (HIGH): skip re-render when props unchanged. useMemo (MEDIUM): cache expensive sorts/filters. useCallback (MEDIUM): stable function refs for memo'd children. Keys (HIGH): use stable IDs, never array index. Avoid inline objects in JSX — new reference every render.

Network Optimizations

staleTime (HIGH): set appropriate cache duration — Infinity for config data, 5min for product lists. Prefetch on hover (MEDIUM): load data before user clicks. Image lazy loading (MEDIUM): native loading="lazy" defers off-screen images.

Advanced Optimizations

Virtualization (HIGH): render only visible items in long lists. useTransition (MEDIUM): mark non-urgent updates so UI stays responsive. Web Workers (LOW): offload CPU-heavy tasks off main thread. Memory leaks (HIGH): always cleanup effects, abort fetches, remove listeners.
