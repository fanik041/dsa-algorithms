📱 Reduce server farm costs for a mobile app
"Use CDN caching for static assets and pre-rendered content. Move heavy animation rendering to the client where possible. Use auto-scaling to spin down idle servers, and consider serverless functions for bursty workloads. Compress assets aggressively and use lazy loading."
⚡ Reduce impact of unreliable third-party APIs
"Circuit breaker pattern — after N failures, stop calling the API and serve cached/stale data. Add a fallback response. Implement retries with exponential backoff for transient failures. Cache recent responses in Redis so an outage doesn't surface to the user."
💰 Estimate resource costs for a video-sharing app
"Need: current DAU/MAU, average video size and watch duration, upload vs view ratio, storage growth rate, bandwidth per stream, CDN costs per GB, current server utilization, and projected user growth %. Then model: storage cost = current GB × growth rate. Bandwidth = DAU × avg watch time × bitrate."
🎵 Single server vs. multiple servers for music streaming
"Single server: simpler ops, lower cost, but single point of failure and limited scalability. Multiple servers: high availability, horizontal scaling, geographic distribution for latency — but adds complexity (load balancing, distributed state, cache consistency). For streaming audio, latency and uptime are critical, so I'd lean multiple servers with a CDN."
💳 Payment system concerns at scale (125k machines)
"Key concerns: network reliability (machines may be offline → need local transaction queuing), idempotency (prevent double charges on retry), PCI-DSS compliance for card data, fraud detection, real-time balance sync vs. eventual consistency trade-off, and rollback/refund handling."

General frameworks to keep handy

REST vs GraphQL REST: simple, cacheable, widely understood. GraphQL: flexible queries, reduces over-fetching, good for complex frontends. Use REST for simple CRUD, GraphQL when clients need varied data shapes.
Cookies vs localStorage Cookies: sent with every request, server-readable, have expiry + HttpOnly flag (safer). localStorage: client-only, larger, persists across sessions. Use cookies for auth tokens, localStorage for UI preferences.

Pattern 1 — 2D Grid Problems (Part 1 + Part 2)
Exact problem type confirmed from Glassdoor
High Priority ▼
Confirmed real questions:
Part 1: Grid of '0' (passable) and '+' (impassable). Return row and column indices that are completely passable.
Part 2: Given a starting corner cell, BFS to find the nearest passable corner cell. Return its coordinates or -1.

Part 1 — Find fully passable rows and columns

def passable_rows_cols(grid):
    rows, cols = len(grid), len(grid[0])

    passable_rows = [
        r for r in range(rows)
        if all(grid[r][c] == '0' for c in range(cols))
    ]
    passable_cols = [
        c for c in range(cols)
        if all(grid[r][c] == '0' for r in range(rows))
    ]
    return passable_rows, passable_cols

# Example
grid = [
  ['0', '0', '+'],
  ['0', '0', '0'],
  ['+', '0', '+'],
]
# passable_rows=[1], passable_cols=[1]

Part 2 — BFS from corner to nearest passable corner

from collections import deque

def nearest_passable_corner(grid, start):
    rows, cols = len(grid), len(grid[0])
    corners = {(0,0), (0,cols-1), (rows-1,0), (rows-1,cols-1)}
    corners.discard(start)  # remove start from targets

    if grid[start[0]][start[1]] == '+':
        return -1

    visited = {start}
    queue = deque([(start, 0)])  # (position, distance)

    while queue:
        (r, c), dist = queue.popleft()

        if (r, c) in corners:
            return (r, c)  # BFS guarantees shortest path

        for dr, dc in [(-1,0),(1,0),(0,-1),(0,1)]:
            nr, nc = r+dr, c+dc
            if (0 <= nr < rows and 0 <= nc < cols
                    and (nr,nc) not in visited
                    and grid[nr][nc] == '0'):
                visited.add((nr,nc))
                queue.append(((nr,nc), dist+1))

    return -1  # no passable corner reachable
Talk through: "I'm using BFS because it finds shortest path on unweighted grids. The corners are (0,0), (0,cols-1), (rows-1,0), (rows-1,cols-1). I'll skip the start corner. First corner I hit in BFS order is the nearest."

Pattern 2 — Board Game / 1D Array + Teleporters
Second confirmed problem type from Glassdoor
High Priority ▼
Confirmed real questions:
Part 1: 1D board, single die roll, list of teleporters (src→dst). Return all distinct positions reachable from start after one roll.
Part 2: Roll die as many times as you want. Can you reach the last square from start? (Reachability via BFS/DFS.)

Part 1 — Positions reachable after one die roll

def reachable_positions(board_size, start, die_sides, teleporters):
    # teleporters: dict {from: to}
    positions = set()

    for roll in range(1, die_sides + 1):
        pos = start + roll
        if pos >= board_size:
            continue  # can't go past the board
        # apply teleporter if one exists
        pos = teleporters.get(pos, pos)
        positions.add(pos)

    return sorted(positions)

# Example: board=10, start=0, die=6, teleporters={3:8}
# Rolls 1-6 → positions 1,2,8(from 3),4,5,6

Part 2 — Can we reach the last square? (BFS reachability)

from collections import deque

def can_reach_end(board_size, start, die_sides, teleporters):
    end = board_size - 1
    visited = {start}
    queue = deque([start])

    while queue:
        pos = queue.popleft()
        if pos == end:
            return True

        for roll in range(1, die_sides + 1):
            nxt = pos + roll
            if nxt >= board_size:
                continue
            nxt = teleporters.get(nxt, nxt)  # apply teleporter
            if nxt not in visited:
                visited.add(nxt)
                queue.append(nxt)

    return False
Key insight to say aloud: "Part 2 is a graph reachability problem — each square is a node, a die roll is an edge. BFS or DFS both work. I'll use BFS. Teleporters are just redirect edges."

How to Communicate During Live Coding
The #1 pass/fail differentiator
Critical ▼
  1. Restate the problem. "So if I understand correctly, I have a grid of '0' and '+', and I need to..." — this catches misunderstandings early.
  2. Clarify edge cases before coding. "Can the grid be empty? Can start be a '+' cell? Is the board 0-indexed?"
  3. State your approach. "I'm going to use BFS because it gives shortest path on unweighted graphs. I'll keep a visited set to avoid cycles."
  4. State complexity upfront. "This will be O(rows × cols) time and space."
  5. Narrate while coding. "I'm defining the four directions here... now I'm checking bounds before adding to the queue..."
  6. Test out loud. Trace through a tiny example manually: "Row 0: all '0', so it passes. Row 1 has a '+', so it fails..."
  7. Discuss trade-offs if time. "An alternative is DFS — same complexity, but BFS gives shortest path which we need here."
