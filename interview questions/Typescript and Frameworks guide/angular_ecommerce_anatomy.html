AppModule / app.config.ts
The entry point
@NgModule
🏛️ The Mayor's Office — Before any shop opens, the mayor registers everything: which buildings exist (declarations), which utilities to connect (imports), which delivery companies to approve (providers), and which building opens first (bootstrap).
AppModule is the root NgModule. Angular reads it at startup to wire the whole app. Angular 17+ apps use standalone components with app.config.ts instead.
@NgModule({ declarations: [AppComponent], imports: [BrowserModule, AppRoutingModule, CoreModule, SharedModule], providers: [], bootstrap: [AppComponent] }) export class AppModule {}
🔒
CoreModule
Singletons only — imported once
@NgModule
🔒 The Security Office — CoreModule houses all app-wide singleton services (Auth, Logger) and HTTP interceptors. It has a guard that throws an error if you try to import it twice.
CoreModule is imported only by AppModule. It provides singleton services and interceptors. The @SkipSelf() + @Optional() constructor guard prevents double imports.
@NgModule({ providers: [ { provide: HTTP_INTERCEPTORS, useClass: AuthInterceptor, multi: true }, { provide: HTTP_INTERCEPTORS, useClass: ErrorInterceptor, multi: true }, ] }) export class CoreModule { constructor(@Optional() @SkipSelf() parent: CoreModule) { if (parent) throw new Error('CoreModule already loaded!'); } }
🧩
SharedModule
Reusable toolkit for all features
@NgModule
🧩 The Hardware Store — SharedModule contains UI components, pipes, and directives that every feature module needs. It re-exports CommonModule and FormsModule. Never put services here — each feature would get its own instance!
SharedModule is imported by every feature module. It declares and exports reusable components, pipes, and directives. Never provide services here.
@NgModule({ declarations: [ HighlightDirective, DebounceClickDirective, IfRoleDirective, DiscountPipe, TruncatePipe, HeaderComponent, SearchBarComponent ], imports: [CommonModule, FormsModule], exports: [ CommonModule, FormsModule, HighlightDirective, DiscountPipe, TruncatePipe, HeaderComponent, SearchBarComponent ] }) export class SharedModule {}
🏪
Services
Business logic + HTTP calls
@Injectable
🏪 The Delivery Company — Services handle all the heavy lifting. Components stay thin — they just call the service and display the result. ProductService fetches products from the API, handles search, filtering, and CRUD operations.
Services contain HTTP calls and business logic. They are decorated with @Injectable({ providedIn: 'root' }) for tree-shakable singletons. Components inject them via the constructor.
@Injectable({ providedIn: 'root' }) export class ProductService { constructor(private http: HttpClient) {} getAll(): Observable<Product[]> { return this.http.get<Product[]>('/api/products'); } }
🛒
CartService (State Store)
BehaviorSubject-based state management
@Injectable
🛒 The Shopping Cart — CartService uses a BehaviorSubject to hold the current cart state. Any component can subscribe to see the cart. When items change, all subscribers are notified instantly.
BehaviorSubject holds the current value and emits it to new subscribers immediately. Use .asObservable() to expose a read-only stream. Mutate state only through service methods.
private itemsSub = new BehaviorSubject<CartItem[]>([]); items$ = this.itemsSub.asObservable(); addItem(product: Product) { const items = [...this.itemsSub.value]; const idx = items.findIndex(i => i.product.id === product.id); if (idx > -1) items[idx].quantity++; else items.push({ product, quantity: 1 }); this.itemsSub.next(items); }
🔐
AuthService
Login, logout, JWT token
@Injectable
🔐 The Bouncer — AuthService manages who's logged in. It stores the JWT token, exposes isLoggedIn$, and route guards use it to protect pages.
AuthService handles login/logout, stores the JWT in localStorage, and exposes an Observable<boolean> for auth state. Route guards check isLoggedIn$ before allowing navigation.
private userSub = new BehaviorSubject<User|null>(null); user$ = this.userSub.asObservable(); isLoggedIn$ = this.user$.pipe(map(u => !!u)); login(email: string, pw: string): Observable<User> { return this.http.post<User>('/api/login', { email, pw }) .pipe(tap(u => this.userSub.next(u))); }
⛓️
HTTP Interceptors
Middleware for every HTTP request
HttpInterceptor
⛓️ The Assembly Line — Every HTTP request passes through interceptors in order: AuthInterceptor adds the token, LoadingInterceptor shows/hides the spinner, ErrorInterceptor catches failures. Like an assembly line that processes every package.
Interceptors implement HttpInterceptor. They are provided with multi: true so Angular chains them in order. Each calls next.handle(req) to pass the request along.
// auth.interceptor.ts intercept(req: HttpRequest<any>, next: HttpHandler) { const token = this.auth.getToken(); if (token) { req = req.clone({ setHeaders: { Authorization: `Bearer ${token}` } }); } return next.handle(req); }

Backend server

Feature Modules
The Franchise
📦 The Franchise — each feature module owns one business domain. Lazy loaded, uses RouterModule.forChild(), imports SharedModule.
Feature modules encapsulate a cohesive set of functionality, enabling lazy loading and clear separation of concerns.
🔀
forRoot() vs forChild()
Router Singleton Guard
🔀 forRoot() sets up the Router + singleton services in AppModule. forChild() only adds routes without re-creating the Router singleton. Using forRoot() in a feature module would create a SECOND Router!

standalone: true
Self-Contained Unit
⚡ Self-Contained Unit — a standalone component doesn't need a NgModule. It lists its own imports right in the @Component decorator. Think of it as a building that comes with its own plumbing and electricity.
Setting standalone: true in the @Component decorator makes the component self-declaring. It no longer needs to be listed in any NgModule's declarations array.
📦
Component imports[]
Personal Toolkit
📦 Personal Toolkit — each standalone component declares which modules, components, pipes, and directives it needs. No more guessing which NgModule provides what.
The imports array in a standalone component lists all dependencies the template needs — other standalone components, pipes, directives, or NgModules like CommonModule.
⚙️
app.config.ts
The Control Panel
⚙️ The Control Panel — replaces AppModule entirely. Contains only providers (no declarations). Uses functional providers like provideRouter(), provideHttpClient().
ApplicationConfig is a simple object with a providers array. It configures application-wide services without needing a class or decorator.
🚀
bootstrapApplication()
Direct Ignition
🚀 Direct Ignition — in main.ts, you bootstrap the root component directly with its config. No AppModule middleman needed.
bootstrapApplication() takes a standalone component and an optional ApplicationConfig. It replaces platformBrowserDynamic().bootstrapModule(AppModule).
🗺️
provideRouter()
Functional Router Setup
🗺️ Functional Router Setup — replaces RouterModule.forRoot(). Cleaner, tree-shakable, and works with standalone.
provideRouter(routes) is a functional API that configures the Angular router. It supports features like withPreloading(), withHashLocation(), etc.
🔌
provideHttpClient()
Functional HTTP Setup
🔌 Functional HTTP Setup — replaces HttpClientModule. withInterceptors() takes functional interceptors instead of class-based ones.
provideHttpClient() enables HttpClient. withInterceptors() accepts an array of functional interceptors — simple functions instead of classes implementing HttpInterceptor.
🔄
Migration Path
Moving Day
🔄 Moving Day — Angular provides ng generate @angular/core:standalone schematic to migrate automatically. You can mix standalone and NgModule-based components during migration.
The migration schematic converts declarations to standalone, updates imports, and eventually removes unnecessary NgModules. Standalone and module-based code can coexist.

import { NgModule } from '@angular/core';
import { RouterModule, Routes, PreloadAllModules } from '@angular/router';

const routes: Routes = [
  { path: '', redirectTo: 'products', pathMatch: 'full' },

  // Lazy load entire feature MODULE
  {
    path: 'products',
    loadChildren: () => import('./features/products/product.module')
      .then(m => m.ProductModule)
  },

  // Lazy load standalone COMPONENT directly
  {
    path: 'cart',
    loadComponent: () => import('./features/cart/cart.component')
      .then(c => c.CartComponent)
  },

  // Lazy with guard — don't even DOWNLOAD if not admin
  {
    path: 'admin',
    loadChildren: () => import('./features/admin/admin.module')
      .then(m => m.AdminModule),
    canLoad: [() => inject(AuthService).hasRole('admin')]
  },

  // Auth module
  {
    path: 'auth',
    loadChildren: () => import('./features/auth/auth.module')
      .then(m => m.AuthModule)
  },

  { path: '**', redirectTo: 'products' }
];

@NgModule({
  imports: [RouterModule.forRoot(routes, {
    preloadingStrategy: PreloadAllModules  // preload after initial load
  })],
  exports: [RouterModule]
})

export class SelectivePreloadStrategy
    implements PreloadingStrategy {

  preload(route: Route, load: () => Observable<any>):
      Observable<any> {
    return route.data?.['preload']
      ? load() : of(null);
  }
}

📦
loadChildren
The Branch Office
📦 The Branch Office — instead of loading the entire mall at once, each department store (feature module) is only downloaded when the customer walks toward it. loadChildren uses dynamic import() to create a separate JS chunk.
loadChildren uses dynamic import() syntax to tell Angular CLI to create a separate bundle for a feature module. The module's code is only downloaded when the user navigates to that route.
⚡
loadComponent
Express Delivery
⚡ Express Delivery — for standalone components, skip the module wrapper entirely. loadComponent imports the component directly. Even lighter than loadChildren.
loadComponent lazily loads a standalone component without needing a wrapping NgModule. It produces a smaller chunk and simpler mental model.
✂️
Code Splitting
Warehouse Sections
✂️ Warehouse Sections — Angular CLI automatically creates a separate JavaScript file (chunk) for each lazy route. The initial page load only downloads the main bundle. Feature bundles are fetched on-demand.
Code splitting is performed by the build tool (webpack/esbuild). Each dynamic import() boundary becomes a separate chunk that is loaded over the network only when needed.
🔄
PreloadAllModules
Stocking the Shelves
🔄 Stocking the Shelves — after the initial page loads and the browser goes idle, Angular silently downloads all lazy modules in the background. The user doesn't wait, but the modules are ready when they navigate.
PreloadAllModules is a built-in PreloadingStrategy. After the app bootstraps, it eagerly fetches all lazy chunks in the background so subsequent navigations are instant.
👁️
@defer (on viewport)
The Shop Window
👁️ The Shop Window — content loads only when the user scrolls it into view. Perfect for reviews, comments, or anything below the fold.
@defer (on viewport) uses an IntersectionObserver internally. When the placeholder element enters the viewport, Angular triggers the dynamic import and swaps in the real content.
💤
@defer (on idle)
Night Shift Restocking
💤 Night Shift Restocking — loads when the browser has nothing else to do. Good for non-critical widgets like recommendations.
@defer (on idle) uses requestIdleCallback to load the deferred block when the browser's main thread is free. This avoids competing with critical rendering work.
👆
@defer (on interaction)
On-Demand Service
👆 On-Demand Service — the component loads only when the user clicks, hovers, or focuses on the placeholder. Great for heavy charts or editors.
@defer (on interaction) listens for click, focus, or keydown events on the placeholder. On first interaction, it triggers the import and replaces the placeholder.
🖼️
@placeholder
The Coming Soon Sign
🖼️ The Coming Soon Sign — shown while the deferred content hasn't started loading. No network request yet.
@placeholder renders lightweight inline content immediately. It acts as the trigger element for viewport/interaction observers. Once loading starts, it transitions to @loading.
⏳
@loading / @error
The Progress Bar
⏳ The Progress Bar — @loading shows while the chunk is downloading. @error shows if the download fails. minimum 300ms prevents flicker.
@loading (minimum Xms) ensures the loading indicator displays for at least X milliseconds, preventing a flash. @error catches network or compilation failures in the deferred chunk.

  @HostBinding('class.out-of-stock')

  @HostListener('mouseenter')
  onMouseEnter() { this.highlighted = true; }

  @HostListener('mouseleave')
  onMouseLeave() { this.highlighted = false; }

    if (changes['product']) {
      console.log('Product changed:', changes['product'].currentValue);

  onAddToCart() {
    this.addToCart.emit(this.product);   // emit event to parent
  }

    this.viewDetail.emit(this.product.id);
  }

@Component Decorator
The Blueprint
🏷️ The Blueprint — @Component tells Angular: this class is a component. It links the TypeScript brain (class) to the HTML face (template) and the CSS clothes (styles). Without it, the class is just plain TypeScript.
@Component is a decorator that marks a class as an Angular component and provides configuration metadata — selector, template, styles, and change detection strategy.
@Component({ selector: 'app-product-card', templateUrl: './product-card.component.html', styleUrls: ['./product-card.component.css'], changeDetection: ChangeDetectionStrategy.OnPush, })
📥
@Input()
The Inbox
📥 The Inbox — @Input makes a property receivable from a parent component. The parent writes [product]="p" and the child receives it. required: true makes Angular error if the parent forgets it.
@Input() decorator marks a class field as an input property, allowing data to flow from parent to child via property binding.
@Input({ required: true }) product!: Product; @Input() showActions = true;
📤
@Output()
The Megaphone
📤 The Megaphone — @Output lets the child shout events at the parent. The child calls emit(), the parent listens with (addToCart)="handler($event)". The EventEmitter is the loudspeaker.
@Output() decorator marks a class field as an output property. Combined with EventEmitter, it enables child-to-parent communication via event binding.
@Output() addToCart = new EventEmitter<Product>(); // child calls: this.addToCart.emit(this.product); // parent hears: (addToCart)="handleAddToCart($event)"

🔍 The Security Camera — @ViewChild lets a parent component directly access a child component or DOM element in its OWN template. Available after ngAfterViewInit. Use a template ref (#myRef) or component type.
@ViewChild queries for the first matching directive, component, or element in the view DOM. The result is set before ngAfterViewInit.

import { Component, Input, OnChanges, OnInit, DoCheck,
         AfterContentInit, AfterContentChecked,
         AfterViewInit, AfterViewChecked, OnDestroy,
         SimpleChanges, ViewChild, ContentChild,
         ElementRef, TemplateRef, ChangeDetectorRef } from '@angular/core';
import { Subject, takeUntil } from 'rxjs';
import { ProductService } from '../product.service';
import { Product } from '../../models/product.model';

@Component({
  selector: 'app-product-detail',
  templateUrl: './product-detail.component.html',
  styleUrls: ['./product-detail.component.css']
})
export class ProductDetailComponent implements
    OnChanges, OnInit, DoCheck,
    AfterContentInit, AfterContentChecked,
    AfterViewInit, AfterViewChecked, OnDestroy {

  @Input() productId!: number;
  @ViewChild('reviewSection') reviewSection!: ElementRef;
  @ContentChild('promoSlot') promoSlot!: TemplateRef<any>;

  product: Product | null = null;
  private destroy$ = new Subject<void>();
  private oldProductId?: number;

  // 0. CONSTRUCTOR — DI happens here. No @Input yet!
  constructor(
    private productService: ProductService,
    private cd: ChangeDetectorRef
  ) {
    console.log('0. constructor — DI only, @Input is undefined');
    // this.productId is undefined here!
  }

  // 1. ngOnChanges — @Input changed (runs BEFORE ngOnInit)
  ngOnChanges(changes: SimpleChanges): void {
    console.log('1. ngOnChanges:', changes);
    // changes['productId'].previousValue
    // changes['productId'].currentValue
    // changes['productId'].firstChange (true on first call)
    if (changes['productId'] && !changes['productId'].firstChange) {
      this.loadProduct(); // re-fetch when parent changes the id
    }
  }

  // 2. ngOnInit — component born, @Input available
  ngOnInit(): void {
    console.log('2. ngOnInit — runs ONCE, @Input ready');
    this.loadProduct();
    // Good for: fetch data, setup subscriptions
    // BAD for: accessing @ViewChild (not ready yet!)
  }

  // 3. ngDoCheck — every change detection cycle
  ngDoCheck(): void {
    // Runs VERY often — keep it lightweight!
    if (this.productId !== this.oldProductId) {
      console.log('3. ngDoCheck — custom check detected change');
      this.oldProductId = this.productId;
    }
  }

  // 4. ngAfterContentInit — projected content ready (ONCE)
  ngAfterContentInit(): void {
    console.log('4. ngAfterContentInit — @ContentChild available');
    if (this.promoSlot) {
      console.log('  Promo slot was projected!');
    }
  }

  // 5. ngAfterContentChecked — after every content check
  ngAfterContentChecked(): void {
    console.log('5. ngAfterContentChecked — content re-checked');
  }

  // 6. ngAfterViewInit — DOM rendered (ONCE)
  ngAfterViewInit(): void {
    console.log('6. ngAfterViewInit — @ViewChild available, DOM ready');
    // NOW you can access @ViewChild
    // this.reviewSection.nativeElement.scrollIntoView();
  }

  // 7. ngAfterViewChecked — after every view check
  ngAfterViewChecked(): void {
    console.log('7. ngAfterViewChecked — view re-checked');
    // WARNING: don't change state here — causes loop!
  }

  // 8. ngOnDestroy — cleanup!
  ngOnDestroy(): void {
    console.log('8. ngOnDestroy — cleanup subscriptions, timers');
    this.destroy$.next();
    this.destroy$.complete();
    // Also: clearInterval, removeEventListener, etc.
  }

  private loadProduct(): void {
    this.productService.getById(this.productId)
      .pipe(takeUntil(this.destroy$))
      .subscribe(p => this.product = p);
  }
}

🔨
constructor
The Delivery Room
🔨 The Delivery Room — the component is born. Angular injects dependencies. But @Input values are NOT available yet — they're still undefined. Never fetch data here.
The constructor is called by JavaScript, not Angular. Use it only for DI. All initialization logic belongs in ngOnInit.
🔄
ngOnChanges
The Mail Notification
🔄 The Mail Notification — fires every time an @Input value changes. Receives a SimpleChanges object with previousValue, currentValue, and firstChange. Runs BEFORE ngOnInit on first render, then on every @Input change.
ngOnChanges(changes: SimpleChanges) { if (changes['productId'] && !changes['productId'].firstChange) { this.loadProduct(); // re-fetch on change } }
🌱
ngOnInit
Opening Day
🌱 Opening Day — runs ONCE after the first ngOnChanges. @Input values are ready. This is where you fetch data, set up subscriptions, and initialize state. The most commonly used hook.
ngOnInit is called once after the first ngOnChanges. Use it for initialization logic that depends on @Input properties.
🔎
ngDoCheck
The Paranoid Inspector
🔎 The Paranoid Inspector — runs on EVERY change detection cycle. Use for custom change detection when Angular can't detect changes (e.g., object mutation). Keep it extremely lightweight — it runs A LOT.
ngDoCheck lets you implement custom change detection. It runs after every change detection cycle, so performance-critical code must be minimal.
📎
ngAfterContentInit
The Mail Arrived
📎 The Mail Arrived — runs ONCE after Angular projects external content into the component via ng-content. @ContentChild is available here, not before.
ngAfterContentInit is called once after Angular fully initializes all content projected into the component via <ng-content>.
📎✓
ngAfterContentChecked
The Mail Re-checked
📎✓ The Mail Re-checked — runs after every change detection check of projected content. Like ngDoCheck but for projected content.
ngAfterContentChecked is called after every check of the component's projected content. Runs after ngAfterContentInit and every subsequent ngDoCheck.
👁️
ngAfterViewInit
The Grand Tour
👁️ The Grand Tour — runs ONCE after the component's view (and child views) are fully rendered. @ViewChild is available here. Perfect for DOM manipulation, chart initialization, or scroll-to.
ngAfterViewInit is called once after Angular fully initializes the component's view and child views. @ViewChild queries are resolved here.
👁️✓
ngAfterViewChecked
The View Re-inspected
👁️✓ The View Re-inspected — runs after every check of the view. WARNING: do NOT change component state here — it causes an infinite loop (ExpressionChangedAfterItHasBeenCheckedError).
ngAfterViewChecked is called after every check of the component's view. Avoid changing bound properties here to prevent infinite change detection loops.
💀
ngOnDestroy
The Demolition
💀 The Demolition — runs when the component is removed. MUST unsubscribe from Observables, clear intervals/timeouts, remove event listeners. Memory leaks happen when you skip this.
ngOnDestroy() { this.destroy$.next(); this.destroy$.complete(); // Also: clearInterval, removeEventListener, etc. }
👨‍👧
Parent vs Child Order
The Family Schedule
👨‍👧 The Family Schedule — Angular creates the parent first, then the children. But children FINISH their view init before the parent. That's why a parent's @ViewChild pointing to a child is available in ngAfterViewInit — the child was fully rendered first.
Lifecycle order: Parent constructor → Parent ngOnInit → Child constructor → Child ngOnInit → Child ngAfterViewInit → Parent ngAfterViewInit.

import { Component, signal, computed, effect, input, output,
         model, ChangeDetectionStrategy } from '@angular/core';
import { Product, CartItem } from '../../models';

@Component({
  selector: 'app-cart',
  standalone: true,
  changeDetection: ChangeDetectionStrategy.OnPush,
  template: `
    <h2>Shopping Cart ({{ itemCount() }} items)</h2>

    @for (item of items(); track item.product.id) {
      <div class="cart-item">
        <span>{{ item.product.name }}</span>
        <span>{{ item.quantity }} x {{ item.product.price | currency }}</span>
        <button (click)="removeItem(item.product.id)">Remove</button>
      </div>
    } @empty {
      <p>Your cart is empty</p>
    }

    <div class="total">
      <strong>Total: {{ total() | currency }}</strong>
      @if (discount()) {
        <span class="discount">-{{ discount() }}% applied!</span>
        <strong>Final: {{ discountedTotal() | currency }}</strong>
      }
    </div>

    <button (click)="onCheckout()" [disabled]="itemCount() === 0">
      Checkout
    </button>
  `
})
export class CartComponent {

  // ── signal() — writable reactive primitive ────────
  items = signal<CartItem[]>([]);

  // ── computed() — derived value, auto-tracks deps ──
  itemCount = computed(() => this.items().length);

  total = computed(() =>
    this.items().reduce((sum, item) =>
      sum + item.product.price * item.quantity, 0
    )
  );

  // ── input() — signal-based @Input replacement ─────
  discount = input<number>(0);   // optional, default 0
  // discount = input.required<number>();  // required variant

  // ── computed using input signal ───────────────────
  discountedTotal = computed(() =>
    this.total() * (1 - this.discount() / 100)
  );

  // ── output() — signal-based @Output replacement ───
  checkout = output<CartItem[]>();

  // ── model() — two-way binding signal ──────────────
  selectedItem = model<CartItem | null>(null);
  // parent uses: [(selectedItem)]="parentSignalOrValue"

  // ── effect() — side effects when signals change ───
  constructor() {
    // Runs whenever items() changes
    effect(() => {
      const currentItems = this.items();
      console.log('Cart updated:', currentItems.length, 'items');
      localStorage.setItem('cart', JSON.stringify(currentItems));
    });

    // Runs whenever total() changes
    effect(() => {
      if (this.total() > 1000) {
        console.log('Big spender! Cart over $1000');
      }
    });
  }

  // ── Updating signals ──────────────────────────────
  addItem(product: Product) {
    this.items.update(items => {
      const existing = items.find(i => i.product.id === product.id);
      if (existing) {
        return items.map(i => i.product.id === product.id
          ? { ...i, quantity: i.quantity + 1 } : i);
      }
      return [...items, { product, quantity: 1 }];
    });
  }

  removeItem(productId: number) {
    this.items.update(items =>
      items.filter(i => i.product.id !== productId)
    );
  }

  onCheckout() {
    this.checkout.emit(this.items());
  }
}

📡
signal()
The Walkie-Talkie
📡 The Walkie-Talkie — a signal holds a value and notifies Angular when it changes. Unlike a variable, Angular knows EXACTLY which template bindings depend on this signal and only updates those specific DOM nodes.
signal() creates a writable reactive primitive. Read with signal(), write with .set(), .update(), or .mutate(). Angular tracks dependencies automatically for fine-grained change detection.
const count = signal(0); count.set(5); count.update(v => v + 1);
🧮
computed()
The Calculator
🧮 The Calculator — a computed signal automatically derives its value from other signals. It auto-tracks which signals it reads and recalculates only when those change. Always read-only.
computed() creates a read-only signal derived from other signals. Angular lazily evaluates and caches the result, recalculating only when a dependency changes.
const total = computed(() => price() * quantity());
⚡
effect()
The Security Camera
⚡ The Security Camera — an effect watches signals and runs a callback whenever they change. Perfect for logging, localStorage sync, or analytics. Automatically cleans up when the component is destroyed.
effect() schedules a callback to run whenever any signal it reads changes. Effects are tied to the injection context and are automatically destroyed with the component.
effect(() => localStorage.setItem('cart', JSON.stringify(items())));
📥
input()
The Signal Inbox
📥 The Signal Inbox — replaces @Input() decorator. Returns a read-only signal that the parent can bind to. input.required<T>() ensures the parent must provide it.
input() creates a signal-based input property. It returns a read-only signal that updates when the parent binding changes. Supports required, default values, and transforms.
discount = input<number>(0); discount = input.required<string>();
📤
output()
The Signal Megaphone
📤 The Signal Megaphone — replaces @Output() + EventEmitter. Cleaner syntax, same behavior.
output() creates an output property without the @Output decorator or EventEmitter. Call .emit() to send events to the parent.
checkout = output<CartItem[]>(); this.checkout.emit(items);
📻
model()
The Signal Walkie-Talkie
📻 The Signal Walkie-Talkie — enables two-way binding with signals. The parent uses [(selectedItem)]='value' and both sides stay in sync.
model() creates a writable signal that supports two-way binding. The parent uses banana-in-a-box syntax [(prop)] and the child can both read and write the value.
selectedItem = model<CartItem | null>(null); // parent: [(selectedItem)]="parentValue"
⚖️
Signals vs BehaviorSubject
Different Tools
⚖️ Different Tools — Signals are for synchronous component state (counters, toggles, form values). BehaviorSubject/RxJS is for async streams (HTTP, WebSocket, complex event chains). Use BOTH in a real app.
Signals provide synchronous, fine-grained reactivity for component state. RxJS/BehaviorSubject excels at async streams and complex event composition. They are complementary, not competing.
// Signals: sync state const count = signal(0); count.set(5); // RxJS: async streams const data$ = http.get('/api').pipe(map(...));
🚫
Zoneless Angular
No More Monkey-Patching
🚫 No More Monkey-Patching — with signals, Angular knows exactly what changed without Zone.js checking the entire tree. provideExperimentalZonelessChangeDetection() in app.config.ts enables this. The future of Angular performance.
Zoneless change detection removes the Zone.js dependency. Angular relies entirely on signals to know when to re-render, eliminating unnecessary change detection cycles.
// app.config.ts export const appConfig = { providers: [ provideExperimentalZonelessChangeDetection() ] };

@Input — Data Down
The Package Delivery
📥 The Package Delivery — parent wraps data in a box ([product]="p") and delivers it to the child's front door. The child just opens the box and uses it. One-way: parent controls the data.

The Customer Feedback Form
📤 The Customer Feedback Form — the child fills out a form (EventEmitter) and hands it back to the parent. The parent decides what to do with it. The child doesn't know or care what the parent does.
@Output() declares an EventEmitter that the child uses to send events to the parent. The parent listens with (event)="handler($event)" syntax. The child is decoupled from the parent's logic.

import { Injectable } from '@angular/core';
import { BehaviorSubject, Observable } from 'rxjs';
import { map } from 'rxjs/operators';
import { CartItem, Product } from '../models';

@Injectable({ providedIn: 'root' })  // singleton — one instance for entire app
export class CartService {

  // Private BehaviorSubject — only this service can .next()
  private itemsSubject = new BehaviorSubject<CartItem[]>([]);

  // Public read-only observables — components subscribe to these
  items$: Observable<CartItem[]> = this.itemsSubject.asObservable();

  itemCount$: Observable<number> = this.items$.pipe(
    map(items => items.reduce((sum, i) => sum + i.quantity, 0))
  );

  total$: Observable<number> = this.items$.pipe(
    map(items => items.reduce((sum, i) =>
      sum + i.product.price * i.quantity, 0))
  );

  // ── Add item ──────────────────────────────────────
  addItem(product: Product): void {
    const current = this.itemsSubject.getValue();
    const existing = current.find(i => i.product.id === product.id);

    if (existing) {
      // Immutable update — new array reference
      this.itemsSubject.next(
        current.map(i => i.product.id === product.id
          ? { ...i, quantity: i.quantity + 1 }
          : i
        )
      );
    } else {
      this.itemsSubject.next([...current, { product, quantity: 1 }]);
    }
  }

  // ── Remove item ───────────────────────────────────
  removeItem(productId: number): void {
    const updated = this.itemsSubject.getValue()
      .filter(i => i.product.id !== productId);
    this.itemsSubject.next(updated);
  }

  // ── Update quantity ───────────────────────────────
  updateQuantity(productId: number, quantity: number): void {
    if (quantity <= 0) return this.removeItem(productId);
    this.itemsSubject.next(
      this.itemsSubject.getValue().map(i =>
        i.product.id === productId ? { ...i, quantity } : i
      )
    );
  }

  // ── Clear cart ────────────────────────────────────
  clear(): void {
    this.itemsSubject.next([]);
  }
}

import { Injectable } from '@angular/core';
import { HttpClient, HttpParams } from '@angular/common/http';
import { Observable } from 'rxjs';
import { map, shareReplay, catchError, retry } from 'rxjs/operators';
import { Product } from '../models';

@Injectable({ providedIn: 'root' })
export class ProductService {

  private apiUrl = '/api/products';
  private cache$?: Observable<Product[]>;

  constructor(private http: HttpClient) {}

  getAll(): Observable<Product[]> {
    // Cache with shareReplay — avoid duplicate HTTP calls
    if (!this.cache$) {
      this.cache$ = this.http.get<Product[]>(this.apiUrl).pipe(
        retry(2),
        shareReplay(1)  // cache the response
      );
    }
    return this.cache$;
  }

  getById(id: number): Observable<Product> {
    return this.http.get<Product>(`${this.apiUrl}/${id}`);
  }

  search(term: string, category?: string): Observable<Product[]> {
    let params = new HttpParams().set('q', term);
    if (category) params = params.set('category', category);
    return this.http.get<Product[]>(`${this.apiUrl}/search`, { params });
  }

  clearCache(): void {
    this.cache$ = undefined;  // force refresh on next getAll()
  }
}

⚖️
Observable vs Promise
Netflix vs Pizza Delivery
⚖️ Netflix vs Pizza Delivery — a Promise is like ordering pizza: it's already being made, you'll get one pizza, and you can't cancel. An Observable is like Netflix: nothing plays until you press play (subscribe), you get multiple episodes (emissions), and you can stop watching anytime (unsubscribe).
A Promise is eager (executes immediately), emits a single value, and cannot be cancelled. An Observable is lazy (executes on subscribe), can emit multiple values over time, and can be cancelled via unsubscribe().
🔥❄️
Hot vs Cold
Live TV vs Netflix
🔥❄️ Live TV vs Netflix — Cold: each subscriber gets their own stream from the start (like Netflix, everyone gets episode 1). Hot: all subscribers share one live stream (like live TV, you join wherever it is). HTTP calls are cold. Subjects are hot. shareReplay(1) converts cold to hot.
A cold Observable creates a new producer for each subscriber (unicast). A hot Observable shares a single producer among all subscribers (multicast). Subjects are hot; HTTP requests are cold. Use shareReplay(1) to multicast a cold Observable.
📢
Subject
The Public Loudspeaker
📢 The Public Loudspeaker — a Subject is both an Observable (others subscribe) AND an Observer (you call .next()). No initial value, no replay. Late subscribers miss past emissions.
A Subject is a special Observable that acts as both observer and observable. It multicasts values to all subscribers but does not replay past emissions. Late subscribers only receive values emitted after they subscribe.
📋
BehaviorSubject
The Whiteboard
📋 The Whiteboard — holds the CURRENT value. New subscribers immediately get the latest value. Perfect for state: current user, cart contents, selected theme. .getValue() for synchronous read.
BehaviorSubject requires an initial value and always holds a current value. New subscribers immediately receive the latest emitted value. Use getValue() for synchronous access. Ideal for representing application state.
📼
ReplaySubject
The DVR
📼 The DVR — records the last N emissions and replays them to new subscribers. ReplaySubject(3) = last 3 messages. Good for notification history, chat logs.
ReplaySubject buffers a configurable number of past emissions and replays them to new subscribers. ReplaySubject(N) keeps the last N values. Useful for caching or providing historical context to late subscribers.
🏁
AsyncSubject
The Final Answer
🏁 The Final Answer — only emits the LAST value and only when complete(). Behaves like a Promise. Use when you only care about the end result of a computation.
AsyncSubject only emits the last value to subscribers, and only after complete() is called. It is functionally similar to a Promise. Use for final computation results where intermediate values are irrelevant.
🏭
Creation Operators
The Factory Floor
🏭 The Factory Floor — of() mocks data, from() converts arrays/promises, interval() creates timers, fromEvent() wraps DOM events. These are the starting points of every Observable chain.
Creation operators produce new Observables. of() emits arguments synchronously; from() converts iterables or Promises; interval()/timer() create time-based streams; fromEvent() wraps DOM events. They are the entry points to reactive pipelines.
🚰
Unsubscribe Patterns
Turning Off the Tap
🚰 Turning Off the Tap — ALWAYS unsubscribe to prevent memory leaks. Best: async pipe (auto). Good: takeUntilDestroyed (v16+). Classic: takeUntil(destroy$). Avoid: manual sub.unsubscribe().
Failing to unsubscribe from long-lived Observables causes memory leaks. The async pipe auto-unsubscribes. takeUntilDestroyed() (v16+) ties lifecycle to DestroyRef. takeUntil(destroy$) is the classic pattern. Manual unsubscribe is error-prone.

🗺️ Routes Configuration

Routes is an array of Route objects that define the URL-to-component mapping for your application.

Each route object can include:

  • path — the URL segment to match
  • component — eagerly loaded component
  • loadComponent — standalone lazy-loaded component
  • loadChildren — lazy-loaded feature module
  • redirectTo — redirect to another path
  • children — nested child routes
  • pathMatch — 'full' or 'prefix'

RouterModule.forRoot(routes) registers routes at the app level. Feature modules use forChild(routes).

Routes is a TypeScript type alias for Route[]. The Router uses a first-match-wins strategy, so route order matters.
🏷️ Route Params vs Query Params

Route params (:id) are required and part of the path: /products/42.

Query params (?key=value) are optional and preserved across navigation: /products?category=shoes&sort=price.

Reading them:

  • paramMap.get('id') — route params
  • queryParamMap.get('category') — query params

Both are available as a one-time snapshot or as a reactive Observable.

Route params define WHAT resource. Query params define HOW to display it (filtering, sorting, pagination).
📂 Nested Routes with children

Child routes render inside the parent component's <router-outlet>, not the root outlet.

This lets you create mini-layouts with shared headers, sidebars, or navigation that persist while child views change.

Feature modules define their own child routes and register them with RouterModule.forChild(routes).

loadChildren lazy-loads an entire feature module (and its child routes) only when the user navigates to that path.

Nested routes are the foundation of feature-based architecture. Each feature module owns its own route subtree.
📤 Named Router Outlets

Multiple <router-outlet> elements can render different components simultaneously.

The primary outlet has no name. Additional outlets use name="xxx".

In route config, set outlet: 'sidebar' to target a named outlet.

URL format: /products(sidebar:help) — parentheses denote the auxiliary route.

Use cases: sidebars, modals, chat panels, help drawers that live independently of the main view.

Named outlets let you compose multiple independent navigation contexts on the same page.

Template: routerLink for static paths, [routerLink] for dynamic arrays. routerLinkActive adds a CSS class when the route is active.

Programmatic: router.navigate(['/products', id]) for absolute paths. Use relative paths with relativeTo option.

queryParamsHandling: 'merge' preserves existing query params when adding new ones.

routerLink is the declarative way (templates). router.navigate() is the imperative way (TypeScript). Both produce the same navigation.
✱️ Wildcard & Redirect

path: '**' is the wildcard route — it catches all unmatched URLs. It must be the LAST route in the array.

redirectTo with pathMatch: 'full' redirects only when the entire remaining URL matches the path exactly.

Route order matters: Angular uses a first-match-wins strategy. More specific routes must come before less specific ones.

Always place the wildcard route last. If placed first, every URL would match it and no other route would ever activate.
📸 Snapshot vs Observable

snapshot.paramMap reads route params once at component creation time.

paramMap (Observable) re-emits whenever params change — for example, navigating from /products/1 to /products/2 without destroying the component.

When to use which:

  • Snapshot: When the component is always destroyed and recreated on navigation (most cases).
  • Observable: When route params might change while the component stays alive (e.g., "Next Product" button).
Angular reuses components by default when only the params change. Use the observable approach if your component needs to react to param changes without being recreated.

export class ProductService {
  private apiUrl = 'https://api.shop.com/products';

  constructor(private http: HttpClient) {}

  getAll(params?: { category?: string; sort?: string })
    : Observable<Product[]> {
    let httpParams = new HttpParams();
    if (params?.category)
      httpParams = httpParams.set('category', params.category);
    if (params?.sort)
      httpParams = httpParams.set('sort', params.sort);
    return this.http.get<Product[]>(this.apiUrl,
      { params: httpParams });
  }

  getById(id: number): Observable<Product> {
    return this.http.get<Product>(`${this.apiUrl}/${id}`);
  }

  create(product: Omit<Product, 'id'>): Observable<Product> {
    return this.http.post<Product>(this.apiUrl, product);
  }

  update(id: number, changes: Partial<Product>)
    : Observable<Product> {
    return this.http.patch<Product>(
      `${this.apiUrl}/${id}`, changes);
  }

  delete(id: number): Observable<void> {
    return this.http.delete<void>(`${this.apiUrl}/${id}`);
  }
}

export const cacheInterceptor: HttpInterceptorFn =
  (req, next) => {
    if (req.method !== 'GET') return next(req);
    const cache = inject(CacheService);
    const cached = cache.get(req.urlWithParams);
    if (cached) return of(cached);
    return next(req).pipe(
      tap(event => {
        if (event instanceof HttpResponse)
          cache.set(req.urlWithParams, event);
      })
    );
  };

Template-Driven Forms

Uses ngModel for two-way binding. #ref="ngModel" creates a template reference to the underlying control. Angular manages the FormControl under the hood automatically.

Good for simple forms like login, search, and contact. Requires FormsModule in your module imports. Validation is done with HTML attributes (required, minlength, email).

Reactive Forms

Explicit FormControl, FormGroup, and FormArray created in TypeScript. You have full control over form structure, validation, and value changes.

Better for complex and dynamic forms. Requires ReactiveFormsModule. Easier to unit test because the form model lives in the component class, not the template.

FormBuilder

Shorthand service for creating reactive forms. fb.group({}) creates a FormGroup. fb.array([]) creates a FormArray. fb.control('') creates a FormControl.

Reduces boilerplate compared to manually calling new FormGroup(), new FormControl(), etc. Inject it via the constructor.

Validators

Built-in validators: required, minLength, maxLength, email, pattern, min, max.

In reactive forms, apply as the second argument to a FormControl: ['', [Validators.required, Validators.email]]. In template-driven forms, use HTML attributes directly on the input element.

Async Validators

Passed as the third argument to a FormControl. Returns Observable<ValidationErrors | null>. Use for server-side checks like username availability or promo code validation.

The control shows a pending state while the async check is in flight. Always debounce to avoid excessive API calls!

Cross-Field Validation

A validator applied at the FormGroup level, not on an individual FormControl. It has access to all child controls in the group.

Useful for rules like "confirm password must match password", "end date must be after start date", etc. Pass in group options: { validators: [fn] }.

FormArray

A dynamic list of form controls. push() adds a new control, removeAt(i) removes one. Perfect for cart items, multiple addresses, or phone numbers.

Access via a getter: get items() { return this.form.get('items') as FormArray; }. Iterate in the template with *ngFor using the getter.

When to Use Which?

Template-driven: simple forms, rapid prototyping, minimal validation logic. Great for login, search, newsletter signup.

Reactive: complex forms, dynamic fields, cross-field validation, extensive testing. Required for FormArray.

Rule of thumb: if the form has 3+ fields or any dynamic behavior, go reactive.

export class IfRoleDirective
  implements OnInit, OnDestroy {
  @Input('appIfRole') requiredRole!: string;
  private hasView = false;
  private sub?: Subscription;

  constructor(
    private templateRef: TemplateRef<any>,
    private viewContainer: ViewContainerRef,
    private authService: AuthService
  ) {}

  ngOnInit() {
    this.sub = this.authService.currentUser$
      .subscribe(user => {
        const match = user?.role === this.requiredRole;
        if (match && !this.hasView) {
          this.viewContainer
            .createEmbeddedView(this.templateRef);
          this.hasView = true;
        } else if (!match && this.hasView) {
          this.viewContainer.clear();
          this.hasView = false;
        }
      });
  }

  ngOnDestroy() { this.sub?.unsubscribe(); }
}

Structural Directives

Structural directives change the DOM structure by adding or removing elements. They are prefixed with * (microsyntax sugar).

Rule: Only ONE structural directive per element.

The built-in structural directives are *ngIf, *ngFor, and *ngSwitch. Under the hood, the * prefix creates an <ng-template> wrapper automatically.

Attribute Directives

Attribute directives change the appearance or behavior of an element without changing the DOM structure. No * prefix.

Built-in examples: ngClass, ngStyle. Custom examples: appHighlight, appDebounceClick.

They are applied as regular HTML attributes on existing elements.

The * Microsyntax

*ngIf="expr" desugars to:
<ng-template [ngIf]="expr">

*ngFor="let x of items" desugars to:
<ng-template ngFor let-x [ngForOf]="items">

Understanding this desugaring is the key to writing custom structural directives — your directive receives the TemplateRef and decides when to stamp it into the DOM.

These are built into the template compiler — no module imports needed. They also enable better type narrowing and compile-time checks.

  transform(price: number,

Built-in Pipes

Angular ships with a rich set of built-in pipes: date, currency, percent, json, slice, async, keyvalue, titlecase, uppercase, lowercase, number.

All are pure (except async). No imports needed with standalone components — they’re in CommonModule.

Pipes transform display values in the template without mutating the underlying data. Think of them as pure formatting functions.

The async Pipe

Auto-subscribes to an Observable or Promise. Auto-unsubscribes on component destroy. Triggers change detection on new value. Prevents memory leaks.

Use it everywhere: {{ data$ | async }}.

Can combine with as syntax: *ngIf="data$ | async as data" — this unwraps the observable and assigns it to a template variable.

The async pipe is technically impure, but Angular optimizes it internally. It’s the one impure pipe you should use freely.

Pure Pipes

Only recalculates when the input value reference changes. Angular caches the previous result.

pure: true is the default — you don’t even need to specify it.

Extremely performant — like a memoized function. The pipe’s transform() only runs when Angular detects a new reference for the input value or arguments.

Always prefer pure pipes over methods in templates. A method runs on every CD cycle; a pure pipe only runs when inputs change.

Impure Pipes

pure: false → runs on every change detection cycle. Needed when the pipe depends on mutable state or when input array contents change without a reference change.

Performance killer — if CD runs 50 times per second, your pipe’s transform() runs 50 times per second regardless of whether the data actually changed.

Avoid unless absolutely necessary. Consider filtering in the component, using computed signals, or creating a new array reference to trigger a pure pipe instead.

Custom Pipe Pattern

Implement the PipeTransform interface. The transform() method receives the value as its first argument plus optional additional args.

Register in module declarations or use standalone: true.

Args are passed with : syntax: {{ val | pipe:arg1:arg2 }}.

Default parameter values work: limit = 50 makes the argument optional in the template.

Pipe Chaining

Pipes compose left to right: {{ price | discount:0.2 | currency:'USD' }}.

Each pipe receives the output of the previous pipe as its input value. Think of it as Unix piping — each stage transforms the data stream.

You can chain as many pipes as needed. Order matters: discount first (returns number), then currency formatting (returns string).

Filter Pipe Anti-Pattern

An impure filter pipe runs O(n) on every CD cycle. At 50 CD cycles/sec with 1000 items = 50,000 iterations/sec. This will destroy your app’s performance.

Better alternatives:

  • Filter in the component (one-time cost)
  • Use a pure pipe with a new array reference
  • Use RxJS operators (map, filter)
  • Use a computed signal (Angular 16+)

🔌 What Zone.js Does

Zone.js monkey-patches ALL async browser APIs: setTimeout, addEventListener, Promise, XHR, fetch, and more.

After any async callback completes, Zone.js notifies Angular. Angular then runs change detection on the entire component tree.

Without Zone.js, Angular wouldn't know when data changes. You'd have to manually call detectChanges() after every async operation.

🔍 Default Strategy

ChangeDetectionStrategy.Default checks the ENTIRE component tree top-down on every async event.

Simple and safe — it never misses a change. But expensive: if you have 500 components, all 500 get checked even if only 1 changed.

Fine for small apps, but problematic for large ones with deep component trees.

⏩ OnPush Strategy

ChangeDetectionStrategy.OnPush only checks when:

  1. @Input reference changes (not mutation!)
  2. DOM event fires in the component
  3. async pipe emits a new value
  4. markForCheck() called manually
  5. Signal value changes (v17+)

Skips the component and its subtree otherwise. Key rule: use immutable data patterns (spread operator, new objects).

🔧 markForCheck() vs detectChanges()

markForCheck(): marks the component and ancestors as dirty. They'll be checked in the NEXT CD cycle. Use for data arriving from outside Angular (WebSocket, third-party lib callbacks).

detectChanges(): runs CD immediately on this component's subtree. Use when you need an instant DOM update (after manual DOM manipulation).

Different tools for different timing needs.

🚫 runOutsideAngular

ngZone.runOutsideAngular() runs code outside Zone.js — Angular doesn't know about it. No change detection is triggered.

Perfect for: animations (requestAnimationFrame), high-frequency intervals, scroll handlers, mouse move tracking.

When you DO need to update the view, call ngZone.run() to re-enter the Angular zone and trigger CD.

✂️ detach() from CD Tree

cd.detach() completely removes the component from the change detection tree.

You must manually call detectChanges() to update the view. Use for rarely-changing components like log viewers or static dashboard widgets.

Maximum performance but maximum responsibility — forget to call detectChanges() and your view goes stale.

⚡ Signals & Zoneless Future

Signals tell Angular exactly which DOM nodes to update. No need to check the entire tree.

provideExperimentalZonelessChangeDetection() removes Zone.js entirely. The result: smaller bundles, faster change detection, and a simpler mental model.

This is the future of Angular reactivity — targeted updates instead of full-tree checks.

Interface vs Type

Interface: for object shapes, supports extends, declaration merging. Preferred for objects and classes in Angular.

Type: for unions, intersections, mapped types. Use type for 'a' | 'b' unions and utility type compositions.

Both can describe object shapes, but interface is the convention for Angular services, components, and models. Interface can be reopened (declaration merging); type cannot.

Use interface for object contracts. Use type for unions, intersections, and utility type compositions.
Generics

Type parameters that make code reusable. ApiResponse<T> works with any data type.

Constraints with extends: T extends { id: number } ensures T has an id property.

Angular uses generics everywhere:

  • Observable<T> — typed streams
  • HttpClient.get<T>() — typed HTTP responses
  • EventEmitter<T> — typed component outputs
Generics let you write one implementation that works safely with many types. The compiler catches mismatches at build time.
Enums vs Union Types

String enums: runtime objects, can iterate over values with Object.values().

Union types: compile-time only, no runtime cost. 'pending' | 'shipped' | 'delivered'.

const enum: inlined at compile time, produces no runtime object.

Many Angular developers prefer union types for simplicity. Use enums when you need runtime access to all values (e.g., populating a dropdown).

Prefer union types for simple string literals. Use enums when you need to iterate over values at runtime.
Utility Types
  • Partial<T> — all optional (for PATCH updates)
  • Omit<T,K> — remove keys (for POST without id)

  • Record<K,V> — key-value map type

Type Guards

Narrow types at runtime so TypeScript knows the exact type in each branch:

  • typeof — for primitives (string, number)
  • instanceof — for class instances
  • in operator — for property existence
  • Custom: item is Product return type

After a guard, TypeScript knows the exact type in that branch. Essential for working with discriminated unions and unknown data.

Type guards bridge runtime checks and compile-time types. They let you safely narrow unknown or union types.
Discriminated Unions

Union types with a common literal field (the discriminant):

{ status: 'loading' } | { status: 'success'; data: T }

switch(state.status) gives exhaustive type narrowing. TypeScript knows data exists only in the 'success' branch.

Perfect for state management patterns — loading states, form states, API responses.

Discriminated unions model states where different variants carry different data. The compiler enforces you handle each case correctly.

Mapped types: { [K in keyof T]: ... } transforms each property of a type.

Conditional types: T extends X ? A : B for type-level if/else logic.

Template literal types: `on${Capitalize<T>}` generates string unions from other string types.

Advanced but powerful for library and framework code. Angular itself uses these patterns internally.

Mapped and conditional types are the building blocks of utility types like Partial, Readonly, and Required.

💾 shareReplay for HTTP Caching

shareReplay(1) caches the last emission. Multiple subscribers share one HTTP call.

Store the Observable (not the value). Reset cache by setting to null.

Perfect for config, feature flags, or any data fetched by multiple components.

Impact: MEDIUM for repeated calls. Eliminates redundant network requests.
🚨 Avoiding Memory Leaks

Unsubscribe patterns: takeUntil(destroy$), takeUntilDestroyed() (v16+), async pipe (auto), take(1) for one-shot.

Clear intervals, remove event listeners in ngOnDestroy.

Leaks = degraded performance over time. Components stay in memory, observables keep emitting.

Impact: HIGH. Silent killer of app performance over time.
